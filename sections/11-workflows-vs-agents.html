Workflows vs Agents

Khi nào dùng deterministic workflow, khi nào để LLM tự quyết? 5 patterns từ Anthropic's "Building effective agents", vai trò của MCP và Agent Skills trong hệ sinh thái 2025-2026, và migration path từ workflow đến agent.

9.1 Định nghĩa: Workflow vs Agent

Workflow

Chuỗi LLM calls được lập trình sẵn (hardcoded DAG). Code quyết định bước nào chạy tiếp theo, không phải LLM.

  • Deterministic control flow
  • LLM chỉ làm task trong từng step
  • Predictable, testable, auditable
  • Số LLM calls cố định

Agent

LLM tự quyết định bước tiếp theo (autonomous loop). LLM điều khiển control flow.

  • Non-deterministic control flow
  • LLM chọn tools và thứ tự dùng
  • Flexible, handles unknowns
  • Số LLM calls variable
≡
Spectrum, không phải binary

Workflow và Agent không phải 2 loại riêng biệt — đây là một spectrum từ "hoàn toàn hardcoded" đến "hoàn toàn autonomous". Hầu hết production systems nằm đâu đó ở giữa.

So sánh trực tiếp

Tiêu chíWorkflowAgent
Predictability Cao — biết trước mọi step Thấp — depends on LLM decisions
Latency Thấp & consistent Variable, thường cao
Cost Thấp & predictable Cao & variable
Reliability at scale Tuyến tính — 99% mỗi step Compound failure — 99%^10 = ~90% cho 10-step agent
Debuggability Dễ — trace từng step Khó — emergent behavior
Flexibility Thấp — cần code change cho new cases Cao — tự adapt
Phù hợp với audit/compliance Tốt — deterministic log Khó — khó reproduce exact path
Task type Well-defined, structured, repeatable Open-ended, variable steps, exploratory

9.2 Khi nào dùng Workflow

Workflow là lựa chọn đúng khi task có thể được decompose thành các steps cố định, và mỗi step có thể được viết thành code rõ ràng.

Dấu hiệu nên dùng Workflow

  • Predictable steps: Biết trước exactly cần làm gì, theo thứ tự nào.
  • High volume / Low latency: Xử lý hàng nghìn requests/giờ với SLA latency nghiêm ngặt.
  • Audit requirement: Cần log chính xác từng step để compliance hoặc debugging.
  • Controlled cost: Cần cost predictable per request — không thể để LLM "quyết định" dùng bao nhiêu tokens.
  • Business rules rõ ràng: "Nếu score > 0.8 thì approve, nếu không thì route to human" — không cần LLM quyết định.

Ví dụ use cases phù hợp với Workflow

  • Document processing pipeline: Load → Extract → Classify → Validate → Store
  • Invoice automation: OCR → Parse fields → Validate → Approve/reject
  • Email triage: Classify → Route → Draft reply (đã templated)
  • Content moderation: Run classifiers → Score → Apply policy rules
  • Data enrichment: Fetch records → Enrich each → Write back

9.3 Khi nào dùng Agent

Agent đúng khi task mở, không biết trước cần bao nhiêu bước, và sự flexible của LLM thực sự tạo value.

Dấu hiệu nên dùng Agent

  • Open-ended task: Không biết trước cần làm gì — task phụ thuộc vào discoveries trong quá trình.
  • Variable steps: Câu hỏi đơn giản cần 2 bước; câu hỏi phức tạp cần 15 bước — cần flexibility.
  • Exploration needed: Cần search, research, hoặc gather info trước khi biết phải làm gì.
  • Complex reasoning: Cần multi-hop, self-correction, hoặc judgment calls mà code không handle được.
  • Latency tolerance: User ok chờ 30–120 giây (hoặc async). Năm 2025-2026, nhiều production agent chạy hoàn toàn async — user nhận kết quả qua notification, không chờ synchronous.

Ví dụ use cases phù hợp với Agent

  • Research assistant: "Tìm hiểu về đối thủ cạnh tranh mới nhất và viết báo cáo"
  • Coding agent: "Fix this bug" — cần read files, understand context, make changes, run tests
  • Complex support: Khách hàng có vấn đề phức tạp cần điều tra nhiều hệ thống
  • Data analysis: "Tại sao revenue Q3 giảm?" — cần query nhiều datasources, reason over results
  • Personal assistant: Multi-step tasks như lên lịch trip, coordinate multiple parties
★
Default: Workflow trước, Agent sau

Anthropic khuyến nghị: "Use the simplest solution that works." Bắt đầu với workflow. Chỉ chuyển sang agent khi workflow rõ ràng không đủ. Agent phức tạp hơn workflow ở mọi dimension: cost, latency, debug, reliability.

Reasoning models (Claude Opus 4, o3, Gemini 2.5 Pro) không loại bỏ nhu cầu workflow. Nghiên cứu 2025 cho thấy ngay cả model có khả năng reasoning mạnh vẫn cần explicit problem structure để tránh constraint violations trong long-horizon tasks. Reasoning tốt hơn giúp mỗi agent step đáng tin cậy hơn — không phải lý do để bỏ qua workflow design.

Decision guide

Câu hỏi task có thể decompose thành steps cố định không?

  • CÓ → Mỗi step có thể viết thành code? → Dùng Workflow
  • CÓ → Step cần LLM judgment? → Workflow + LLM trong từng step
  • KHÔNG → Steps thay đổi tùy discoveries? → Cân nhắc Agent
  • KHÔNG → Latency <2s critical? → Reconsider, có thể redesign thành workflow
  • KHÔNG → Latency flexible, task open-ended? → Dùng Agent

9.4 Anthropic's 5 Building Blocks

Từ bài viết "Building effective agents" của Anthropic, 5 patterns cốt lõi để compose LLM applications — từ đơn giản đến phức tạp. Hiểu 5 patterns này = hiểu phần lớn production AI systems.

① Prompt Chaining

Chuỗi LLM calls tuần tự: output của call N là input của call N+1. Mỗi step thực hiện một tác vụ nhỏ, tập trung hơn là cố nhồi tất cả vào một mega-prompt.

Khi nào dùng: Task có thể chia thành N sub-tasks độc lập, tuần tự. Mỗi step cần full attention của LLM. Gate giữa steps để catch errors sớm.

Ví dụ: Draft marketing copy → Translate → Check brand compliance → Format for channel.

② Routing

Phân loại input và gửi đến pipeline phù hợp. LLM (hoặc classifier model) đóng vai "router", quyết định handler nào xử lý request.

Khi nào dùng: Nhiều loại input cần xử lý khác nhau. Router giúp mỗi downstream pipeline được tối ưu cho task cụ thể thay vì cố handle everything.

Ví dụ: Customer support router (billing / technical / general / escalate), content pipeline (long-form / short-form / social), language detection (vi / en / other).

③ Parallelization

Chạy nhiều LLM calls song song rồi aggregate kết quả. Hai sub-patterns: Sectioning (chia document lớn, process parallel) và Voting (run same task N lần, pick majority).

Sectioning: Chia document lớn thành sections, process song song, merge. Good for: summarization, extraction từ long docs.

Voting: Run same prompt N lần với temperature >0, pick majority answer. Good for: high-stakes classification, fact-checking.

④ Orchestrator-Workers

Orchestrator LLM nhận task lớn, dynamically plan và dispatch sub-tasks cho worker LLMs/agents. Orchestrator synthesize final answer. (Xem chi tiết diagram ở §8.7 — đây là multi-agent topology phổ biến nhất.)

Khi nào dùng: Task lớn cần nhiều loại expertise. Workers có thể specialize (research vs. writing vs. analysis). Orchestrator không cần biết details — chỉ delegate.

Ví dụ: Deep research agent, complex software development (plan → code → test → review), business intelligence report generation.

≡
MCP: chuẩn kết nối tools cho Orchestrator-Workers (2024-2026)

Anthropic's Model Context Protocol (MCP) — ra mắt 11/2024, được OpenAI và Google adopt vào 2025, donate cho Linux Foundation cuối 2025 — trở thành de facto standard để agent gọi tools và kết nối data sources. Claude hiện có 75+ connectors qua MCP directory.

Code Execution với MCP (2025): Thay vì pass tool outputs trực tiếp qua model context, pattern mới yêu cầu model viết code để gọi MCP tools rồi execute. Kết quả: workflow từng tốn ~150,000 tokens chỉ còn ~2,000 tokens — giảm 98.7% token usage. Cực kỳ relevant cho Orchestrator-Workers pattern ở production scale.

⑤ Evaluator-Optimizer

Generator tạo output; Evaluator critique; Generator refine; loop đến khi đạt quality threshold. Variant của reflection (§8.6) nhưng ở cấp workflow — có thể là 2 separate LLM calls, không chỉ prompting tricks.

Khi nào dùng: Output quality cần exceed threshold (legal copy, marketing, code review). Evaluator có thể là LLM judge, automated tests, hoặc human-in-the-loop.

Ví dụ: Code generation (generate → run tests → fix failures), translation (translate → back-translate → compare), SEO content (write → SEO score → rewrite).

★
Tất cả 5 patterns có thể combine

Real-world systems thường kết hợp: Routing → Orchestrator-Workers (với mỗi worker là Prompt Chaining) → Evaluator-Optimizer trên output cuối. Hiểu 5 patterns này cho phép compose bất kỳ AI system phức tạp nào.

≡
Agent Skills — reusable capability modules (cuối 2025 / 2026)

Anthropic ra mắt Agent Skills vào cuối 2025 và mở làm open standard vào 3/2026. Skills là các reusable capability modules được invoke tự động qua Claude.ai, Claude Code, và API — không cần maintain logic riêng cho từng surface.

Thay vì nhồi toàn bộ logic vào system prompt, teams đóng gói behavior thành Skills có thể version, inspect, và govern. Kết quả: token overhead giảm (chỉ load context cần thiết), audit dễ hơn, và behavior nhất quán hơn across interactions. OpenAI, Cursor đang xem xét adopt cùng spec — tín hiệu rằng Skills có thể trở thành standard tương tự MCP.

9.5 Hybrid Approach — Workflow với Agent Steps bên trong

Không cần chọn hoàn toàn workflow hay agent. Pattern phổ biến nhất trong production: deterministic workflow ở outer level, agent capabilities ở specific steps cần flexibility.

Ví dụ: Invoice Processing Hybrid

# Outer workflow: deterministic steps
def process_invoice(invoice_file):

    # Step 1: Deterministic — extract raw text
    raw_text = ocr.extract(invoice_file)

    # Step 2: LLM — structured extraction (1 call, deterministic output)
    extracted = llm.extract_structured(raw_text, schema=InvoiceSchema)

    # Step 3: Deterministic — validate against business rules
    if not validator.validate(extracted):
        return route_to_human(extracted)

    # Step 4: AGENT step — complex reconciliation
    # This step needs tools because outcome varies: might need to
    # query vendor DB, check PO system, handle discrepancies
    reconciled = reconciliation_agent.run(
        invoice=extracted,
        tools=[query_vendor_db, query_po_system, flag_discrepancy]
    )

    # Step 5: Deterministic — route based on reconciliation result
    if reconciled.auto_approvable:
        return auto_approve(reconciled)
    else:
        return route_for_manual_review(reconciled)

Nguyên tắc Hybrid design

  • Default deterministic: Mọi step là code trừ khi có lý do rõ ràng để dùng LLM.
  • Isolate agent steps: Agent step có scope rõ ràng — nhận input X, trả về output Y trong contract được define.
  • Gate sau agent step: Validate output của agent step trước khi tiếp tục workflow.
  • Timeout mỗi agent step: Agent step có thể hang — luôn có timeout riêng.

9.6 Migration Path: Workflow → Agent

Không nên "just build an agent" từ đầu. Path chuẩn là tăng dần autonomy khi trust được xây dựng.

Các giai đoạn migration

Stage 1: LLM-augmented Workflow

Thêm LLM calls vào workflow có sẵn cho các steps cần language understanding. Control flow vẫn hardcoded. Low risk

Stage 2: LLM với limited tool use

Cho LLM gọi tools trong phạm vi hạn chế (read-only, 2–3 tools, bounded iterations). Workflow vẫn dictate high-level flow. Medium risk

Stage 3: Agent trong bounded context

Agent có autonomy trong một specific domain với guardrails rõ ràng. Human-in-the-loop cho critical actions. Higher risk

Stage 4: Full autonomous agent

Agent handle toàn bộ task end-to-end, chỉ escalate exceptions. Chỉ đạt stage này sau khi có extensive evaluation data. High risk — only with data

Checklist trước khi tăng autonomy

  • Eval set > 100 real-world cases, passing rate > 95%
  • Failure modes được catalog và có mitigation
  • Observability đầy đủ — có thể trace mọi decision
  • Rollback plan rõ ràng — có thể revert về stage trước trong 1 giờ
  • Cost và latency impact đã được measure và accepted
  • Legal/compliance review đã sign off
⚠
Autonomy is earned, not given

Tăng autonomy từng bước nhỏ. Mỗi lần tăng: monitor kỹ trong 2 tuần, rollback ngay nếu có anomaly. Agent fail ở production cost rất cao — cả về money lẫn user trust.

Tóm tắt

  1. Workflow = deterministic graph (code controls flow); Agent = autonomous loop (LLM controls flow).
  2. Workflow tốt hơn khi: task predictable, high-volume, low-latency, audit-needed, cost-sensitive.
  3. Agent tốt hơn khi: task open-ended, variable steps, cần exploration và adaptation.
  4. 5 Anthropic patterns: Prompt Chaining (sequential), Routing (classify+dispatch), Parallelization (sectioning/voting), Orchestrator-Workers (hierarchical), Evaluator-Optimizer (generate+critique+refine).
  5. Hybrid là common case: deterministic outer workflow + agent steps ở chỗ cần flexibility.
  6. Migration path: Stage 1 (LLM in workflow) → Stage 2 (limited tools) → Stage 3 (bounded agent) → Stage 4 (full autonomous). Tăng từng bước sau khi có data.
  7. MCP (2024-2026): Model Context Protocol là standard kết nối tools cho agent. Pattern "code execution với MCP" giảm 98.7% token usage so với truyền tool outputs qua context.
  8. Agent Skills (2025-2026): Reusable capability modules — version được, audit được, token-efficient — becoming open standard tương tự MCP.
  9. Compound reliability: Agent 10 bước với 99% per-step chỉ đạt ~90% end-to-end success. Workflow giữ reliability tuyến tính. Càng nhiều steps → càng cần kiểm soát tốt từng step.
  10. Default: workflow. Reasoning models mạnh hơn không thay thế workflow design — chúng giúp mỗi step đáng tin cậy hơn. Chứng minh workflow không đủ trước khi chuyển sang agent.
